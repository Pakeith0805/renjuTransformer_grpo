## 0. 原論文（Allis 1994）の Threat-Space Search

出典: L.V. Allis, H.J. van den Herik, M.P.H. Huntjens, "Go-Moku and Threat-Space Search" (1994) /
Allis 博士論文 "Searching for Solutions in Games and Artificial Intelligence" (1994)。

### 脅威（threat）の定義
各脅威は **gain square（攻め側が打つ1マス）** と **cost squares（その脅威を防ぐために
盤上が占められる必要があるマスの集合）** で特徴づける。種類:

| 脅威 | 説明 | 防御側の応手 |
|---|---|---|
| five | 五完成＝勝ち | なし |
| four / straight four | 四（あと1手で五）。達四(両端開き)は受け不能 | 五点を1点に強制（達四は受け不能=勝ち） |
| (open) three | 活三（あと1手で達四） | **複数の応手がある**（三を殺すマス群） |
| broken three など | 跳び三等 | 同上 |

### winning threat sequence（必勝脅威列）
各脅威が防御側の応手を強制し、最終的に **二重脅威（同時に2つの脅威＝受け切れない）** に
到達する脅威の列。攻め側は **脅威手の空間だけ** を探索する（分岐が激減）。

### conservative defender（保守的防御 ＝ 健全性の要）
脅威列が本当に勝ちかを判定するとき、**防御側に「それまでの全脅威の cost squares を一度に
占めさせる」** という最も防御的な仮定で攻め側がなお勝てるかを見る。これで防御側の各応手を
個別展開せずに健全に判定できる。**ただし防御側が自分の脅威（カウンター）で手番を奪える場合は
別途考慮が要る**（＝後述の現行VCFの穴に直結）。

### dependency-based search / 脅威の組み合わせ
脅威は依存関係を持つ（ある脅威の gain square が別の脅威を可能にする）。Allis は依存グラフ上で
脅威の組み合わせを効率的に探索する。

### VCF / VCT は特殊ケース
- **VCF（四追い, Victory by Continuous Fours）**: 脅威を **四のみ** に限定した脅威列探索。
  四は受けが一意（五点1点）なので防御分岐が単純。
- **VCT（四三追い, Victory by Continuous Threats）**: 四＋三。三は受けが複数＝本格的な木探索。

## 1. 現行実装（2026-06 時点）＝ VCF only

`solve_vcf_recursive`（[mcts.cpp:1300]）:
1. 即五（`is_winning_move`）を最優先。
2. 脅威手は **四を作る手のみ**（`creates_four_threat`）。三は探索に未使用
   （`count_open_three_directions` は禁手判定専用）。
3. 四を打った後の五完成マスを数え、**≥2 → 受け不能で勝ち / 1 → その1点を防御側に強制ブロックさせ再帰**。
4. 連珠対応: 黒の禁手を攻め/受け両側で考慮。
5. 有界: 深さ12（四を12回）＋20万ノードで打ち切り→ `-1`。

→ 原論文の「四・受け一意・連続四」という**最小サブセット**のみ。三/conservative defender/
依存探索は未実装。プロジェクト内の呼称「TSS」は実体 VCF。

## 2. 既知の不健全性（2026-06-28 ハーネスで検出）

`scripts/verify_tss_soundness.py` を現行VCFに対して実行: **120局中 ~3%（4件）が偽陽性**
（VCFが必勝と返すが防御側に逃げ手がある）。分類の結果 **「相手に即五あり」は 0/4** ＝
単純な見落としではない。

**原因（濃厚）: 防御側カウンター脅威の無視。**
`solve_vcf_recursive` は「四→防御は五点を受けるだけ」と仮定し、再帰では攻め側の次の四しか
探さない（[mcts.cpp:1369]）。だが防御側が**受けの代わりに自分の四を作って手番を奪える**と、
攻め側のVCFが途切れる。VCF はこの防御カウンターを一切チェックしていない。

→ 原論文の枠組み（conservative defender ＋ 防御側自身の脅威の考慮）は**この穴を構造的に塞ぐ**。
「忠実な実装」にすれば偽陽性は原理的に消えるはず。

影響: 報酬の own-VCF+1 が約3%は実際は必勝でない手に加点 / 評価 attack ラベルが約3%誤り /
推論TSSが稀に「必勝のつもりが負ける手」を打つ。小さいが全系に乗る。

## 3. 連珠固有の差分（原論文はフリースタイル五目）

- **黒の禁手**（三三・四四・長連）。脅威生成・防御応手・勝敗判定すべてで考慮が必要。
  現行VCFは対応済み（`is_forbidden_for_black`）。VCT/TSS でも踏襲する。
- **達四/五の扱い**: 五は禁手の対象外（`is_winning_move` で長連のみ除外）。
- 盤は 15x15、候補は石から距離2以内に限定（`neighbor_candidates`）。

## 4. 段階的実装プラン（2026-06-28 再構成）

各段の完了条件は **`verify_tss_soundness.py` で false_positive=0**。

**再整理の根拠**: `has_five_or_more` は連続5個ベースでクリーン。**四は受けが一意**なので、
P0後のVCFは原理的にほぼ健全（残存0.8%は四VCFの稀エッジ or 検証器アーティファクトのどちらか、未確定）。
**conservative defender の本質は「三（受けが複数）」を健全に扱う機構**なので、旧P1とP2は実質
一体（conservative defender＝三を扱うための装置）。よって：

- **P0.x: 四VCF残存(0.8%)の決着（小）**
  - まず**正体を機械的に確定**（後述の計装）。
    - solve_vcf主張手順の各「受け」が本当に唯一の合法防御か（防御側に同等以上の代替が無いか）を出力。
    - 検証器側も硬く: attacker_wins の失敗理由を「防御が即五で勝つ/solve_vcfが-1/予算切れ」に分類。
  - 結果次第:
    - **四VCFの実バグ** → 該当ロジック(終端 or forced-block 仮定)をピンポイント修正。
    - **検証器のアーティファクト**（局所候補 local_empty の取りこぼし等）→ 検証器を修正し VCF は P0 で確定。
  - 完了条件: 四VCF単体で false_positive=0（回帰盤面＝2026-06-28のFP局面を含む）。
- **P1（=旧P1+P2 一体）: conservative defender による VCT（四＋三）**
  - **脅威モデル**: 各脅威を gain square（攻めが打つ1マス）＋ cost squares（防御が脅威を消すために
    占有しうるマス集合）で表現。四: cost=完成1点(受け一意)。達四: それ自体勝ち。
    活三/跳び三: 達四化を1手で狙える形。cost=複数（両端・割り点）。
  - **conservative defender（健全性の核）**: 脅威列を評価する際、防御側に**それまでの全脅威の
    cost squares を一括占有**させた最防御盤で、攻めがなお winning threat（達四/二重脅威）に届くかを見る。
    実防御は毎手1石＝cost squares の部分集合しか置けないので、最防御盤で勝てれば任意の実防御に勝てる＝健全。
    （四は cost が一意なので最防御盤＝実盤で一致し、現行の forced-block 再帰と等価。三で初めて効く。）
  - **防御側カウンターの一級市民化**: 防御側が自分の四（攻めに応手を強いる）を持つ枝を明示的に扱う。
  - **AND-OR 探索**: OR=攻めの脅威手列挙 / AND=防御の cost squares 応手列挙。ノード予算と depth で有界化。
  - 完了条件: VCT で false_positive=0。`test_tss_imitation` の活三系カテゴリが射程に入る。
- **P2: 依存探索 / 脅威の組み合わせ（性能が要るとき）**
  - 二重脅威検出を依存グラフで効率化。まずは素朴な AND-OR で正しさ優先、遅ければ此処で最適化。

**全段共通の鉄則**: 報酬・評価オラクル・推論MCTSは**同一の新ソルバー**を呼ぶ（整合を単一実装で担保）。
各段 C++改修 → リモート再ビルド → ハーネス(false_positive=0) → 接続、の順。検証器自身も
回帰盤面で硬くしてから次段へ。

各段で **C++ 改修 → DLL/so 再ビルド（実行コンテナ glibc 一致が必須, 既知の罠）→ ハーネス →
報酬/評価/推論に接続** の順。報酬・評価・推論は**同一の新ソルバーを呼ぶ**ことで整合を保つ
（現行 VCF が単一実装で整合を担保しているのと同じ構造）。

### 2026-06-28 — P0 実装（防御カウンター修正）
- `mcts.cpp` にヘルパ `defender_has_immediate_win(board, defender)` を追加。
- `solve_vcf_recursive` と `solve_vcf_recursive_path` の「攻めの四を置いた直後」に挿入:

### 2026-06-28 — 残存FPの真因判明: 検証ツールのバグ(ソルバーは正しかった)
**逆転**: 新solve_vctもfours-onlyオラクルも同じ盤面でFP。Pythonに移植して5ノードで再現し追跡 →
solve_vct の“勝ち手順”終端が **「攻-相手禁手で受け不能(9,8)」**。決定的チェック:
- 終端で受け点 **(8,8) は黒の禁手**(`is_forbidden_for_black=True`)、かつ (8,8) に白を打つと白五。
- ＝白(9,8)で四→五点(8,8)、黒は禁手で(8,8)に受けられない→白勝ち。**黒の禁手を突く本物の連珠勝ち**。

→ **solve_vcf(P0後)もsolve_vctも正しい。** バグは**検証ツール側**: `bf_forced_win` と `attacker_wins` が
「黒は禁手点に受けられない」を考慮せず、黒に禁手で受けさせて「勝ち無し」と誤判定していた(偽警報)。
- P0で消えた3件は本物のバグ(防御側即五の見落とし)。残り1件だけが、この禁手防御の検証ツール不備。
**修正**: `legal_defender_moves`(黒は禁手除外)を追加。attacker_wins とオラクルの受け列挙・強制ブロック判定に適用
(唯一の受けが黒禁手なら攻め勝ち)。これで偽警報が消える。
**教訓**: 検証器も連珠ルール(黒禁手)を完全実装しないと、正しいソルバーを誤って偽陽性と裁く。
独立オラクル比較→Python移植→トレース→決定的チェック、の順で真因に到達できた。
